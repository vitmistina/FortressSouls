# B-025: v0.1 release review

Status: Ready
Parent backlog: `docs/backlog/v0.1-backlog.md#b-025`
Product specification: `docs/specs/fortress-souls-v0.1.spec.md`
Primary module: Whole-repository review
Risk class: Critical
Recommended implementation model: Release/critical safety review tier (GPT-5.5)
Recommended reasoning level: xhigh
Recommended review model: Independent release review tier (GPT-5.5 xhigh)
Human checkpoint: Required

## Observable Outcome

A dated release review records exact evidence, unresolved findings, and a human-readable go/no-go recommendation for v0.1 safety, scope, contracts, fake mode, optional live modes, observability, documentation, and checks.

## Why This Slice Exists

The release needs an independent evidence-based gate, not more implementation. The review must verify that backend listing, browser selection, by-ID snapshots, and session identity binding agree across code, tests, and documentation.

## Context To Load

Required context:

- `AGENTS.md`, this mini-spec, and B-025
- all accepted ADRs
- product spec, backlog, mini-spec completion reports, architecture, and public contracts
- completed B-024 documentation/runbook pass
- `.agents/agents/reviewer.agent.md`
- `.agents/prompts/review-backlog-item.prompt.md`
- all canonical scripts/runbooks and final implementation/test manifests

Conditional context:

- DFHack/provider manual evidence only when it is dated, environment-scoped, and explicitly retained.
- git history/CI evidence only if available; do not infer results.

## Existing State To Inspect

At planning time application implementation and `docs/reviews/` do not exist. At review time start from clean `git status`, inventory the actual tree/contracts/tests, and compare every backlog status/acceptance criterion with evidence.

## In Scope

- Perform review only and create `docs/reviews/v0.1-release-review.md`.
- Record revision/date/environment, exact validation commands/results, acceptance matrix, scope/safety review, dependency/advisory evidence, fake-mode browser result, optional live-provider/DFHack evidence, observability/redaction review, and docs/runbook usability.
- Search for persistence, game mutation, generic command/process surfaces, model tools, streaming, secret/content telemetry, unsafe paths, provider DTO leakage, and module cycles.
- Classify findings by severity with file/line evidence and owners/follow-up.
- Give explicit `GO`, `NO-GO`, or `CONDITIONAL NO-GO` with human sign-off checkpoint.

## Out Of Scope

- No feature/code fixes, broad documentation rewrite, dependency upgrade, status laundering, tag/commit/push, live-service claim without evidence, or acceptance of known critical/high safety defects.

## Boundaries And Invariants

- Review is independent from implementation assumptions and cites current evidence.
- No game mutation, arbitrary command execution, model tool access, persistence, streaming, or microservice split may be waived for v0.1.
- Secrets, prompts, responses, conversation, dwarf names, raw DFHack output, and private paths do not enter the review artifact.
- A missing/unrun check is reported as missing evidence, never as pass.
- Backlog status changes require acceptance evidence and are separate explicit edits.

## Implementation Slices

### Slice 1: Automated and structural evidence

- Intended behavior: all canonical checks and safety/architecture searches have recorded results.
- Likely files or modules touched: review document only.
- Test-first evidence: documentation/review exception; establish the review matrix before running commands.
- Completion evidence: exact command/result/limitations recorded for every gate.

### Slice 2: User/operational evidence

- Intended behavior: fake journey and runbooks are verified; live modes are accurately labeled.
- Likely files or modules touched: review document only.
- Test-first evidence: run the actual documented workflows rather than asserting prose.
- Completion evidence: observable outcomes and failures recorded without sensitive content.

### Slice 3: Findings and release decision

- Intended behavior: findings lead, acceptance coverage is explicit, and go/no-go follows severity/evidence.
- Likely files or modules touched: `docs/reviews/v0.1-release-review.md`.
- Test-first evidence: independently cross-check findings against root safety and ADRs.
- Completion evidence: human can make the release decision without rerunning analysis.

## Acceptance Criteria

- [ ] Review proves the browser owns selection from the backend list and no DF UI cursor state affects behavior.
- [ ] Review artifact records revision, environment, exact commands, results, and unverified/manual evidence.
- [ ] Every v0.1 definition-of-done and backlog acceptance area maps to evidence or a finding.
- [ ] Safety, redaction, architecture, dependencies, fake browser journey, provider, DFHack, observability, and docs are explicitly reviewed.
- [ ] Findings are severity-ordered with precise file/line evidence and no prohibited content.
- [ ] Recommendation is explicit and human approval is required before tag/release.

## Test Strategy

This is a review artifact, so red-green-refactor is not applicable. Correctness evidence is execution of canonical checks, browser/manual workflows, static searches, contract comparison, diff review, and an independent second-model review. Do not modify implementation during the review; file findings for anything beyond tiny explicitly authorized documentation corrections.

## Observability And Failure Behaviour

Review required spans/metrics/events, correlation, cardinality, redaction, and safe user errors through tests and local evidence. Record signal names/outcomes only, never captured content or secrets.

## Validation

Run the canonical commands that exist at release time, expected to include:

```powershell
.\scripts\format.ps1
.\scripts\test.ps1
.\scripts\check.ps1
git diff --check
git status --short
```

Also run the focused fake browser smoke command and documented manual checks when prerequisites are available. Record every exact result in the review.

## Stop Conditions

- Stop with NO-GO if authoritative product, safety, or architecture conflicts remain.
- Stop with NO-GO for mutation, arbitrary command execution, model tools, secret/content leakage, persistence, or unexplained failing required checks.
- Stop rather than implementing fixes inside the review item.

## Completion Report

Report: 1. outcome and review decision; 2. changed review files; 3. validation commands and results; 4. known limitations, unverified assumptions, remaining manual checks, and required human sign-off.
